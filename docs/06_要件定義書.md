## 1. プロジェクト概要

### 1.1 ビジネスゴール

| # | ゴール |
|---|--------|
| 1 | 店舗での接客オペレーションの改善（バーコード提示による会員特定の迅速化） |
| 2 | EC顧客のユーザビリティ向上（定期情報・購入履歴の自己解決） |
| 3 | 会員ランク制度の体験価値向上（ボルネオの聖地に近づく世界観演出） |

### 1.2 ユーザー種別

| 種別 | 説明 |
|------|------|
| 未登録ユーザー（非会員） | アプリをダウンロードするだけで利用開始。仮会員証・店舗ポイント・チェックイン・店舗一覧・ブランドコンテンツを利用可能 |
| BI-SU 会員（EC購入者・店舗来店者） | ログイン＋楽楽連携により、定期情報・購入履歴・ランク制度・全会員機能を利用 |

- **ハイブリッド方式**（v2.1決定）: ログイン不要で基本機能が使える。連携は任意とし、連携時に会員機能を解放する

### 1.3 プラットフォーム

| 項目 | 内容 |
|------|------|
| フレームワーク | Flutter |
| iOS | 16以上（仮。OP-22で確定） |
| Android | 10（API 29）以上（仮。OP-22で確定） |
| 画面基準 | スマホ縦持ち優先。幅 320〜430pt 級で崩れないこと（モック 390x844 基準） |

### 1.4 進め方

1. **HTMLモックアップフェーズ**（完了）--- デザイン・体験を確定
2. **設計・定義フェーズ**（v2.1進行中）--- 14エピック・70ストーリー構成
3. **本実装フェーズ**（次工程）--- Flutter による iOS / Android アプリ化

### 1.5 アプリの性格

- **閲覧専用ミラー**で確定（yuka回答 2026-06-12「見るだけ」）
- 会員情報・定期・履歴は閲覧のみ。変更操作はECマイページへ誘導（＋変更締切日の表示）
- **ハイブリッド方式を採用**（6/18会議決定）: ログインなしで基本機能が使える。ログイン＋楽楽連携は任意

### 1.6 認証方式

- **MIRANEST App準拠**で確定（yuka回答 2026-06-17）
- メールアドレスまたは電話番号 → 認証コード（メール/SMS）→ 認証完了 → 楽楽のEC顧客データと紐付け
- 楽楽に未登録の場合はカスタマークリエイトAPIで顧客を新規作成して紐付け
- Amazon Pay認証を代替手段として調査中（OP-31）

## 2. 機能要件

> FR-xx IDから領域別（AREA-xx）IDへの読み替え表は本章末の対応表を参照。
> 出典凡例 --- ✅ yuka指示 / 🌐 公式準拠 / 💭 仮置き・推奨案

### 2.1 認証・ログイン（AUTH）

| ID | 要件 | 優先度 |
|----|------|--------|
| AUTH-01 | **認証・ログイン**: MIRANEST App準拠の認証フロー。メールアドレスまたは電話番号入力 → 認証コード（メール/SMS）受信 → コード入力で認証完了 → 楽楽のEC顧客データとメールアドレスで照合・自動紐付け。楽楽に未登録の場合はカスタマークリエイトAPIで顧客を新規作成。セッショントークン管理、リフレッシュトークンによる自動再認証。機種変更時はメールアドレス/電話番号で再認証（端末間データ転送不要）。Amazon Pay認証の代替利用を調査中（OP-31） | Must |
| AUTH-02 | **初回オンボーディング**: 初回起動時に3〜4画面のオンボーディング（アプリ概要・ランク制度説明・通知許可・利用開始）。ログイン/楽楽連携はスキップ可能（ハイブリッド方式。スキップしても基本機能が使える旨を明示）。連携メリット（定期情報・購入履歴・ランク特典・ポイント優遇率）を案内。2回目以降は非表示 | Must |
| AUTH-03 | **アカウント削除**: App Store審査要件として必須。多段階確認フロー（設定最下部テキストリンク → 注意事項表示 → パスワード/認証コード再認証 → 最終確認ダイアログ → 72時間クーリングオフ期間 → メール確認通知） | Must |
| AUTH-04 | **ハイブリッド方式（非ログインベース）**: ログインなしで基本機能が使えるアプリ設計。未ログイン時利用可: 仮会員証・店舗ポイント・チェックイン・店舗一覧・ブランドコンテンツ・多言語切替・問い合わせ。ログイン必須: 定期情報・購入履歴・ランク制度・通知設定。未ログイン時はログイン必須機能をグレーアウト表示し、連携メリットを案内して誘導 | Must |
| AUTH-05 | **楽楽EC連携**: 認証完了後にメールアドレスで楽楽の顧客データを照合。一致すれば購入履歴・ランク・ポイントを引き継ぎ。未登録の場合はカスタマークリエイトAPIで顧客を新規作成。S3データ連携から楽楽API連携に全面移行（OP-26）。データ移行計画が必要 | Must |

### 2.2 会員ランク・ステータス（MEMBER）

| ID | 要件 | 優先度 |
|----|------|--------|
| MEMBER-01 | **ホーム＝ステータス画面**: ランクによって「背景画像・ステータステキスト・BI-SUロゴ」が切り替わる。ポイント残高・有効期限・次ランクまでの進捗バー・年間購入累計を表示 | Must |
| MEMBER-02 | **会員ランク5段階**: MEMBER / SILVER / GOLD / PLATINUM / DIAMOND。英語表記は公式準拠。基本ランクの正式名称は「MEMBER」で確定 | Must |
| MEMBER-03 | **ランク制度連動**: 表示ランク＝前年確定の公式ステータス。進捗バーは「今年（1/1〜12/31）の購入累計→翌年ステータス」の訴求。画面注記の文言例:「ステータスは前年1年間（1/1〜12/31）のご購入累計額で決まり、当年1年間適用されます」 | Must |
| MEMBER-04 | **3Dロゴエンブレム**: ロゴマークの3D表現。ランクによって色味が変わる | Should |

### 2.3 会員証・店頭提示（CARD）

| ID | 要件 | 優先度 |
|----|------|--------|
| CARD-01 | **会員証バーコード**: ホーム左上に会員証バーコード提示ボタン。タップで読み取り可能なバーコードをモーダル表示（v2.1: QR→バーコード先行。全店舗スキャナ棚卸し後に最終決定→OP-27）。店舗側はスマレジでバーコードを読み取り、会員IDを転記する運用。オフラインでも表示できること（直営店はB1F等の地下にあり圏外があり得る）。輝度自動最大化 | Must |
| CARD-02 | **仮会員証**: 未登録（未ログイン）ユーザーに仮のバーコードを発行し、店舗ポイントの付与のみを可能にする。ログイン＋楽楽連携後に仮会員証で貯めたポイントを本アカウントに引き継ぐ。仮会員証はデバイスローカルに識別子を保持 | Must |

### 2.4 店舗ポイント（POINT）

| ID | 要件 | 優先度 |
|----|------|--------|
| POINT-01 | **店舗ポイント**: EC（BASE）ポイントとは独立した店舗専用ポイント制度。ログインなしでも付与・照会が可能。来店購入時にバーコード提示でポイント付与。チェックインでもポイント付与。楽楽連携済みの場合はポイント付与率を優遇。用途（クーポン交換等）・失効条件・重複防止は後続企画で確定（OP-28）。ホーム画面に店舗ポイント残高を表示 | Must |

### 2.5 チェックイン（CHECKIN）

| ID | 要件 | 優先度 |
|----|------|--------|
| CHECKIN-01 | **チェックイン（来店記録）**: 購入を伴わない来店を記録する機能。チェックインにより店舗ポイントを付与。方式はTBD（店内QR読取・位置情報・Bluetooth等を候補として検討中→OP-29）。不正利用防止の仕組みが必須（同日複数回制限等）。ログインなしでも利用可能 | Should |

### 2.6 起動演出（LAUNCH）

| ID | 要件 | 優先度 |
|----|------|--------|
| LAUNCH-01 | **起動演出**: ランク別クローズアップ演出（上空→島→ジャングル→洞窟入口→洞窟内→巣）。リッチで躍動感。毎回起動時に再生で確定。スキップ常設とし、NFR-03のtime-to-barcode（3秒）はスキップ経由で担保 | Must |
| LAUNCH-02 | **起動演出（動画）**: 一般ランクはAI生成動画。ボルネオ島の全景俯瞰で着地（後半カットはyuka指示）。最終フレーム＝ホーム背景。reduced-motion有効時は静止画フォールバック、動画未DL時はプレースホルダー+プログレス表示 | Should |

### 2.7 定期情報（SUB）

| ID | 要件 | 優先度 |
|----|------|--------|
| SUB-01 | **定期情報**: 商品画像/商品名/ステータス（注文継続中・解約済み）/次回お届け予定日/サイクル/個数/金額を表示。閲覧専用のため、変更・スキップ・解約はECマイページへの誘導ボタン＋変更締切日の表示で対応。ステータスの中間状態（一時停止・決済失敗等）の要否はEC仕様で確定（OP-24） | Must |

### 2.8 購入履歴（PURCH）

## 3. 非機能要件

### 3.1 パフォーマンス

| ID | 区分 | 要件 | SLA/目標 |
|----|------|------|----------|
| NFR-01 | 対応OS | iOS / Android（Flutter） | iOS 16以上 / Android 10（API 29）以上（仮。OP-22で確定） |
| NFR-03 | 演出品質 | 起動演出＋バーコード提示までの導線 | 起動演出は約5.5秒以内＋スキップ可。起動からバーコード提示まで目標3秒以内（スキップ・バーコード直行） |
| NFR-08 | 画面対応 | スマホ縦持ち優先 | 画面幅320〜430pt級で崩れないこと（モック390x844基準） |

### 3.2 セキュリティ

| ID | 要件 |
|----|------|
| NFR-09 | 認証トークンのセキュアストレージ保存（Keychain/Keystore）。通信はすべてTLS 1.2以上。バーコードに個人情報を直接埋め込まない。ログアウト時のローカルキャッシュ完全消去。バックグラウンド遷移時の画面マスク（個人情報保護）。楽楽API認証の安全性確保（APIキーの秘匿・通信経路の暗号化・アクセス権限の最小化） |

### 3.3 オフライン・フォールバック

| ID | 要件 |
|----|------|
| NFR-02 | モックフェーズ: 単一HTML・オフラインで全機能動作（約7.2MB）。本実装では参考扱い |
| NFR-05 | Flutter前提のフォールバック: (1) reduced-motion有効時 → 静止画で即ホーム遷移 (2) 動画未ダウンロード時 → プレースホルダー+プログレス表示 (3) バッテリー節約モード時 → 静止画フォールバック。通信不可・API障害・メンテナンス時の画面挙動も要定義 |

### 3.4 アクセシビリティ

| ID | 要件 |
|----|------|
| NFR-06 | reduced-motion対応（NFR-05連動）。OSの文字サイズ設定への追従（200%まで）。写真背景上のテキストコントラスト基準（WCAG AA目安）。顧客層に高年齢層を含むため |

### 3.5 ブランド表現

| ID | 要件 |
|----|------|
| NFR-04 | 公式トンマナ（アイボリー x シャンパンゴールド・明朝系）準拠。ランクカラーは公式準拠（PLATINUM=青系・DIAMOND=深紅系） |

### 3.6 データ鮮度

| ID | 要件 |
|----|------|
| NFR-07 | 表示データの更新頻度と「いつ時点の情報か」の画面表示を定義（アプリ表示とEC・店頭レシートの正本宣言を含む。OP-12） |

## 5. 制約・前提

| # | 前提 |

## 6. 会員ランク制度

> 出典: BI-SU公式サイト membership ページ（2026-06-11取得）＋ yukaさん提供のステータス振り分けルール（2026-06-12）

### 6.1 集計・適用ルール（確定）

| 項目 | ルール |
|------|--------|
| 集計期間 | 当年 1/1〜12/31 の買い物累計額 |

## 7. 運用要件

> アプリ単体ではなく「店舗・CS・マーケの業務」と対で成立する要件群。ここが白紙のまま開発に入らないこと。

### 7.1 店舗オペレーション（OP-01, OP-27）

- 店舗POS=スマレジ。アプリのバーコードをスマレジで読み取り、会員IDを転記する運用（v2.1: QR→バーコード先行。全店舗スキャナ棚卸し後に最終決定→OP-27）
- 店舗ポイント運用: 会員証提示時にスマレジ経由でポイント付与。仮会員証でもポイント付与可能。チェックインによるポイント付与の店舗側フローも要設計
- 残課題: スマレジのコード読取仕様、購入データ→EC会員データの連携経路、スマホを出せない会員への代替照会手段、店舗スタッフ向け手順書、全店舗のスキャナ環境棚卸し

### 7.2 会員ライフサイクル（OP-06）

- 認証方式=MIRANEST App準拠で確定。メール/電話番号→認証コード→楽楽EC顧客データと紐付け
- ハイブリッド方式により非会員もアプリ利用可能（仮会員証→ログイン時にポイント引き継ぎ）
- 残課題: 認証コード有効期限・リトライ制限、セッション有効期間、退会時のデータ処理範囲、複数端末・家族利用の可否、楽楽カスタマークリエイトAPI可否（OP-26）

### 7.3 データの正本とCS対応（OP-12）

- アプリ表示とEC・店頭レシートが食い違った時の正本宣言とCSエスカレーションフロー
- 店頭購入が履歴へ反映されるまでの時間の明示

### 7.4 コンテンツ運用（OP-21, OP-17）

- 店舗情報・期間限定POP UPの更新フロー（誰が・いつ・どうやって）
- 公式制度・営業時間の改定を検知して表示と整合させる定期点検（四半期目安）

### 7.5 計測・KPI（OP-13）

- アプリの狙い（確定）: (1) 店舗での接客オペレーションの改善 (2) EC顧客のユーザビリティ向上
- 候補KPI: バーコード提示数、スマレジ転記の所要時間、店舗ポイント付与件数 / 定期・履歴の自己解決率、アプリ経由のECマイページ遷移数
- 計測イベントは実装時に仕込む必要があるため、本実装の要件として確定する

### 7.6 店舗ポイント運用（OP-28, OP-29）

- **付与ルール**: 来店購入時（バーコード提示）およびチェックイン時にポイント付与。付与率はログイン状態・楽楽連携状況で変動（連携済みが優遇）
- **利用ルール**: ポイントの用途（クーポン交換・割引・特典等）は後続企画で確定
- **失効ルール**: 有効期限・失効条件は後続で確定
- **不正防止**: 同日複数回チェックインの制限、仮会員証の同一デバイス制限、ポイント付与の監査ログ
- **運用フロー**: 店舗スタッフのポイント手動修正・取消の手順、ポイント残高の問い合わせ対応

### 7.7 多言語コンテンツ運用（OP-30）

- 対応言語: 日本語・English（v1スコープ）
- 翻訳体制: 翻訳の担当者・品質基準・レビュー体制は後続で確定
- コンテンツ管理: 言語別コンテンツの更新フロー（ブランドコンテンツ・店舗情報・利用規約等）
- 祇園店舗PR制約: アプリ内文言・同意文言が祇園店舗のPR制約に準拠すること（OP-32）

## 付録A: 認証方式の確定（2026-06-17）

yukaさんの回答により認証方式がMIRANEST App準拠で確定。

**確定した認証フロー**（v2.1: ステップ4を楽楽に変更）:

1. ユーザーがメールアドレスまたは電話番号を入力
2. 認証コードがメールまたはSMSで送信される
3. ユーザーが認証コードを入力して認証完了
4. 認証完了後、楽楽のEC顧客データとメールアドレスで照合・自動紐付け（未登録の場合はカスタマークリエイトで新規作成）
5. 以降はトークンベースのセッション管理で自動再認証

**影響範囲**: OP-06を★★に降格 / AUTH-01を正式定義 / NFR-09（セキュリティ）を定義 / SCR-010, SCR-011を定義
