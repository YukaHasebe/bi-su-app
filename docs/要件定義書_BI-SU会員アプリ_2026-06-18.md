## 1. 背景・目的

- BI-SU（エムスタイルジャパンが運営する、人向けのツバメの巣 健康食品・化粧品ブランド）の**会員向けスマホアプリ**を作る。Claude Code の練習を兼ねたプロジェクト。✅
- 進め方は2段階: **①HTMLモックアップでデザイン・体験を確定 → ②Flutterで本実装**。✅
- アプリの中核コンセプトは「**会員ランクが上がるほど、ツバメの巣の源流（ボルネオの聖地）に近づいていく**」体験演出。✅

## 2. 対象ユーザー・利用シーン

- BI-SU の会員（EC購入者・店舗来店者）。
- 主な利用シーン: ①店舗での会員証（QR）提示、②定期購入の確認、③注文履歴の確認、④店舗を探して来店、⑤自分のランク・ポイントの確認。
- ⚠ 利用シーン①は**店舗側の読み取りオペレーション（誰が・何で読み・どう処理するか）とセット**で初めて成立する。アプリ単体では完結しない（→OP-01）。

## 5. 非機能要件

| ID | 区分 | 要件 |
|---|---|---|
| NFR-01 | 対応OS | iOS / Android（Flutter）。💭仮案: iOS 16以上 / Android 10（API 29）以上 →OP-22で確定 |
| NFR-02 | 配布性（モック） | 単一HTML・オフラインで全機能動作（現行 約7.2MB）※本実装フェーズでは参考扱い |
| NFR-03 | 演出品質 | 起動演出は約5.5秒以内＋スキップ可。💭**起動からQR提示まで目標3秒以内**の導線（スキップ・QR直行）を確保（レジ前渋滞の防止。→OP-07）。計測: コールドスタートからQRモーダル表示完了までの実測時間 |
| NFR-04 | ブランド表現 | 公式トンマナ（アイボリー×シャンパンゴールド・明朝系）準拠。ランクカラーは公式準拠（PLATINUM=青系・DIAMOND=深紅系） |
| NFR-05 | 耐障害性・フォールバック | **Flutter（iOS 16+/Android 10+）前提のため、SVGフォールバックは不要**。代わりに以下のフォールバック条件を定義: ①**reduced-motion有効時** → 静止画（演出最終フレーム）で即ホーム遷移 ②**動画未ダウンロード時** → プレースホルダー画像＋プログレス表示 ③**バッテリー節約モード時** → 静止画フォールバック（動画再生を抑止）。💭本実装では**通信不可・API障害・メンテナンス時の画面挙動**も定義必須 |
| NFR-06 | アクセシビリティ | reduced-motion対応（上記NFR-05と連動）。本実装ではOSの文字サイズ設定への追従（200%まで）と、写真背景上のテキストコントラスト基準（WCAG AA目安）を追加（顧客層に高年齢層を含むため） |
| NFR-07 | データ鮮度 | 💭表示データの更新頻度と「いつ時点の情報か」の画面表示を定義（アプリ表示とEC・店頭レシートの正本宣言を含む。→OP-12） |
| NFR-08 | 画面対応 | 💭スマホ縦持ち優先。画面幅320〜430pt級で崩れないこと（モックは390×844基準） |
| **NFR-09** | **セキュリティ** 🆕v2.0 | 認証トークンのセキュアストレージ保存（Keychain/Keystore）。通信はすべてTLS 1.2以上。QRコードに個人情報を直接埋め込まない。ログアウト時のローカルキャッシュ完全消去。バックグラウンド遷移時の画面マスク（個人情報保護） |

## 7. 運用要件（本実装までに定義が必要な業務要件）

アプリ単体ではなく「店舗・CS・マーケの業務」と対で成立する要件群。**ここが白紙のまま開発に入らないこと。**

### 7.1 店舗オペレーション（→OP-01）
- ✅ **店舗POS=スマレジ。アプリのQRをスマレジで読み取り、会員IDを転記する運用**（yuka回答 2026-06-12）
- 残課題: ①スマレジのコード読取仕様（QR読み取りで会員検索/ID入力できる設定・コード体系）の確認 ②スマレジの購入データ→EC会員データ（年間購入金額・購入履歴）への連携経路 ③スマホを出せない会員への代替照会手段（氏名＋電話番号等） ④店舗スタッフ向け手順書
- ※店頭購入が会員に紐付かない場合、FR-08の「店頭購入」表示も年間購入金額の正確性も成立しない

### 7.2 会員ライフサイクル（→OP-06）
- ✅ **認証方式 = MIRANEST App準拠で確定**（yuka回答 2026-06-17）:
  - メールアドレスまたは電話番号で認証コード（メール/SMS）を受け取り、認証完了後にS3上のEC顧客データと紐付け
  - 機種変更時はメールアドレス/電話番号で再認証するだけでデータが復元される（端末間のデータ転送は不要）
- 残課題: ①認証コードの有効期限・リトライ制限の仕様 ②セッション有効期間 ③退会（アカウント削除）時のデータ処理範囲 ④複数端末・家族利用の可否

### 7.3 データの正本とCS対応（→OP-12）
- アプリ表示とEC・店頭レシートが食い違った時の正本宣言とCSエスカレーションフロー
- 店頭購入が履歴へ反映されるまでの時間の明示

### 7.4 コンテンツ運用（→OP-21・OP-17）
- 店舗情報・期間限定POP UPの更新フロー（誰が・いつ・どうやって）
- 公式制度・営業時間の改定を検知して表示と整合させる定期点検（四半期目安 💭）

### 7.5 計測・KPI（→OP-13）
- ✅ **アプリの狙い（確定・yuka回答 2026-06-12）: ①店舗での接客オペレーションの改善 ②EC顧客のユーザビリティ向上**
- これに沿った候補KPI 💭: ①QR提示数・スマレジ転記の所要時間（接客効率）／②定期・履歴の自己解決率（CS問い合わせの減少）・アプリ経由のECマイページ遷移数
