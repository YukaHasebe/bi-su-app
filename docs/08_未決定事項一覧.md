## 2. ブロッカー（★★★）— 決まらないと主要機能が作れない

### OP-05｜ポイント制度のシステム仕様

| 項目 | 内容 |
|---|---|
| 状態 | **部分確定** |
| 決定者 | 未定 |
| 影響画面 | SCR-100（ホーム）, TBD-501（店舗ポイント残高） |
| 影響エピック | EP-3（ランク・ステータス）, EP-5（店舗ポイント） |

**確定済み（2026-06-24）:**
- BI-SUポイント（EC連動）は**楽楽EC の API から直接取得・表示**する。アプリ側での計算は不要
- モックの「年間金額×還元率」式は使わない

**残課題:**
- 店舗ポイントは**制度自体が未策定**。できたときに実装できる設計にしておく
- BI-SUポイントの有効期限・付与タイミングの表示項目は API 仕様に依存
- 配送料・割引・ポイント利用の表示有無（OP-15 と連動）

### OP-24｜EC基盤 API の仕様・認証方式（楽楽移行）

| 項目 | 内容 |
|---|---|
| 状態 | **未決定** |
| 決定者 | 会社（システム） |
| 影響画面 | SCR-200（定期情報）, SCR-300（購入履歴） |
| 影響エピック | EP-7（定期情報）, EP-8（購入履歴）, EP-12（楽楽連携） |

**わかっていること:**
- 顧客情報・注文履歴・定期情報は **Snowflake**（バックエンドAPI経由）から取得する方針が確定（2026-06-24）

**必要な情報:**
- Snowflake → バックエンドAPI のデータ取得仕様
- 定期コース情報・購入履歴のレスポンス形式
- 列挙値（経路・注文ステータス・定期ステータス）の実値一覧
