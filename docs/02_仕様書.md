### 4.3 動画演出の共通仕様
- 着地時に`canvas`へ最終フレームをキャプチャ（JPEG品質0.86）→ ホーム背景に適用
  - キャプチャ条件: `readyState>=2 && currentTime>0.5 && 要素が接続中 && 結果が20KB超`（黒画像の誤保存防止）。失敗時はSVGシーン背景へフォールバック
- `play()`失敗時は320ms×n間隔で最大3回リトライ → 不可ならSVGフォールバックへ着地（iOS省電力モード等対策）
- スキップ時も着地フレームまでシークしてからキャプチャ
- 推奨生成プロンプト指針: 縦9:16 / 尺3〜6秒 / **最後は静止気味の安定ショットで終える**

## 5. ホーム 仕様

画面ID: SCR-100

レイアウト（上から）:
1. **バーコードボタン（左上・ガラス質感）** 🔄v2.1 / 中央にBI-SUロゴ（白版PNG）
2. **3Dロゴエンブレム**（§6）
3. `MEMBERSHIP STATUS` ラベル → ランク英字（疎組み明朝・40px）→ 日本語ステータス名 → ロケーション表記 → **「ステータス詳細」リンク（→ SCR-102 CLUB MEMBERS PROGRAM）**
4. 会員名（ダミー）
5. ガラスカード①: 年間ご購入金額 + 次ランク進捗バー + 「翌年 {次ランク} まであと ¥xxx」
6. ガラスカード②: ポイント残高 + 還元率チップ「100円 = Npt」

✅本実装: ホーム画面のポイント残高（ガラスカード②）には有効期限も表示する（DoD S-3.1参照）。**BI-SUポイントは楽楽EC APIから直接取得・表示（アプリ側で計算しない）**。有効期限等の表示項目はAPI仕様に依存（OP-05残課題: 店舗ポイントは制度未策定）

モックの計算式（**架空式・ダミー値**）:
- 進捗% = (年間購入金額 − 現ランク閾値) ÷ (次ランク閾値 − 現ランク閾値) ×100（最小4%・最大100%）
- DIAMONDは進捗100%固定＋お礼メッセージ
- ポイント残高 = 年間購入金額 ÷ 100 × 還元率

✅本実装のランク・進捗表示ロジック（確定ルール受領 2026-06-12 により具体化）:
- **表示ランク** = 前年（N−1年）1/1〜12/31の購入累計で確定した**当年（N年）の公式ステータス**
- **進捗バー** = 当年（N年）1/1からの購入累計 ÷ 次ランク閾値（30万/50万/100万/200万円）→ **翌年（N+1年）ステータスの見込み**訴求
  - 当年累計が現ステータスの閾値未満でも負値表示にしない（0%起点の単純な「翌年に向けた積み上げ」として表示。モックの「現ランク閾値からの相対式」は使わない）
  - 当年実績が0円でも進捗バーは0%で表示（非表示にしない）
- 画面注記例💭: 「ステータスは前年1年間（1/1〜12/31）のご購入累計額で決まり、当年1年間適用されます」
- ステータスは毎年再判定（下位へ戻ることもある）。演出は現ステータス準拠で出し分け（OP-23決定）

⚠残課題:
- 累計額の細部（税込/税抜・返品減算・店頭購入の包含・ポイント利用分）はOP-03残課題の確定待ち
- 「BI-SUポイント残高」は楽楽EC APIから直接取得・表示（アプリ側計算なし、確定）。店舗ポイントは制度未策定のためOP-05残課題。モックの「年間金額×還元率」式は使わない
- ランク特典（ギフト・招待）の表示は現状なし → 扱いはOP-10

### 未ログイン時のホーム表示 🆕v2.1

ハイブリッド方式により、未ログイン状態でもホーム画面を表示する。

- **ブランドビジュアル**: BI-SUの世界観を伝える静的ビジュアル（起動演出のMEMBER演出 or ブランドキービジュアル）
- **連携案内**: 「ログインして会員特典を確認」の案内カード（タップでSCR-010へ）
- **店舗ポイント残高**: 仮会員証で貯めたポイントがあれば表示（EP-5参照）
- **非表示項目**: ランク表示・進捗バー・BI-SUポイント残高・3Dロゴのランク色（ログイン後に解放）
- バーコードボタンは仮会員証のバーコードを表示（§7参照）

背景: ランクの到達シーン（MEMBERは動画最終フレーム写真）＋ 可読性スクリム ＋ 16秒のスロー寄り引き（1回再生・負荷対策）

## 6. 3Dロゴエンブレム 仕様（logo3d.js）

- ライブラリ: Three.js r160（`three.min.js` 654KBを同梱）

## 12. データ構造（モックの実装 + ⚠本実装向け注記）

```
RANKS{id}: { en, jp, rate, color, soft, threshold, next, sceneA, sceneB,
             video?, videoEnd?, place, annual(ダミー) }
MEMBER:    { name, initials, no, mail, since }
SUBSCRIPTIONS[]: { name, img, status('active'|'cancelled'),
                   next?, last?, cycle, qty, price }
ORDERS[]:  { date(出荷日), no, ch(経路), chSub?(店舗名), kind(種別),
             name, img, qty, price }
STORES[]:  { id, name, en, type(直営店|取扱店|期間限定), img?, imgClass?,
             dist, city, addr, hours, closed, tel?, period?,
             access[], desc, plan?(見取り図キー), planNote? }
```

⚠本実装（このままモデルクラス化しないこと。B1の規約とセットで再設計する）:
1. **ORDERSは1注文=1商品の簡略構造** → 実装は「注文ヘッダ（注文番号・日付・経路・合計）＋明細items[]（商品・単価・数量・行金額）」の2階層にする
2. `price`の意味がダミーでは曖昧（qty:2, price:8100 は行合計）→ B1で「単価」「行合計」「注文合計」を別フィールドとして定義
3. BI-SUは**食品(税率8%)と化粧品(10%)が混在** → 税表示の計算根拠（税込のみ表示か、税率内訳を持つか）をEC仕様に合わせる
4. `STORES`に **lat/lng（座標）を追加**する（距離計算・地図表示の前提）
5. 配送料・割引・ポイント利用の表示有無をOP-05/OP-15と合わせて決定
6. 🆕v2.1 **店舗ポイントモデルを追加**: `STORE_POINTS{ balance, history[], provisional_id? }`。仮会員証のポイントも含む。詳細は §B8
7. 🆕v2.1 **チェックインモデルを追加**: `CHECKINS[]{ store_id, timestamp, method, point_awarded }`。詳細は §B9
8. 🔄v2.1 会員データの紐付け元は**楽楽EC**（旧S3）。楽楽APIのメール照合・カスタマークリエイトを経由してデータ取得（§11.5・§B1参照）

## B3. セキュリティ・プライバシー指針 💭 🔄v2.1

1. **会員証バーコードの設計** 🔄v2.1: 🟢 認証方式＝MIRANEST App準拠（メール/電話番号→認証コード→楽楽EC紐付け）で確定（2026-06-17）。店舗読み取り=スマレジ・ID転記運用が決定（2026-06-12）。バーコードの中身はスマレジ読取仕様に合わせ、**ランク・特典情報はバーコードに含めない**（スマレジ/楽楽EC側データを正とする）。なりすまし緩和は店頭本人確認＋画面の動的要素💭。署名付き短期トークン化はスマレジ仕様と両立する場合の上位案として保持（OP-01残課題とセットで確定）
2. **会員番号の画面表示**: 常時フル表示は避け、マスク表示＋タップで開示を検討（肩越し盗み見対策）
3. **スクリーンショット**: バーコード画面のスクショ抑止（Android `FLAG_SECURE` 等）の要否を検討。iOSは技術的に抑止不可のためトークン短命化で対処 🔄v2.1
4. **個人情報**: 氏名・メール・購入履歴を扱うため、端末内キャッシュの暗号化と、プライバシーポリシーへの明記（位置情報・計測を含む）が必要（OP-11）
5. **モック専用のデバッグ機能はリリースビルドから完全除去**（B4）
6. **認証トークン管理**: セッショントークンはセキュアストレージ（iOS Keychain / Android Keystore）に保存。リフレッシュトークンで自動再認証。ログアウト・アカウント削除時はローカルキャッシュの完全消去。バックグラウンド遷移時の画面マスク（NFR-09参照）
7. **楽楽API認証セキュリティ** 🆕v2.1: 楽楽EC APIへのアクセスはサーバーサイド経由とし、APIキー/シークレットをアプリに埋め込まない。メール照合・カスタマークリエイトのリクエストはバックエンドAPIを中継。レート制限・リトライポリシーを設計（OP-26で仕様確定後に詳細化）
8. **仮会員証のセキュリティ** 🆕v2.1: 仮IDはデバイスローカルで生成。仮IDからの個人情報逆引きは不可とする設計。本登録時の紐付け処理はサーバーサイドで検証（同一仮IDの重複紐付け防止）

## B8. 店舗ポイント仕様 🆕v2.1

> EP-5 に対応。BI-SUポイントとは独立した店舗専用ポイントシステム。

### 基本方針

- **BI-SUポイントとは独立**: BI-SUポイント（楽楽EC連動）とは別体系。店舗での購買・来店行動に対して付与
- **ログイン不要で利用可能**: 仮会員証（§B7）でもポイント付与・照会が可能
- **楽楽連携で優遇**: ログインし楽楽EC連携を完了した会員はポイント付与率が優遇される（EP-5 S-5.5）

### データフロー

```
[付与]
来店/購入 → スタッフがバーコード読取 → スマレジで会員ID特定 → ポイント付与API → 残高更新

[照会]
アプリ起動 → ホーム画面 or ポイント画面でポイント残高・履歴を表示

[利用]
会員がポイント利用を申告 → スタッフがバーコード読取 → ポイント消費API → クーポン/特典と交換
```

### ポイントモデル（💭仮置き）

| フィールド | 型 | 説明 |
|---|---|---|
| member_id | string | 会員ID（正会員ID or 仮ID） |
| balance | integer | 現在の残高 |
| history[] | array | 付与・利用の履歴 |
| history[].type | enum | `earn_purchase` / `earn_checkin` / `earn_bonus` / `redeem` |
| history[].amount | integer | ポイント数（付与は正・利用は負） |
| history[].store_id | string | 対象店舗 |
| history[].timestamp | ISO 8601 | 日時 |
| history[].description | string | 付与理由の表示テキスト |

### 要決定事項（OP-28）

- 付与条件（購入金額あたりのポイント数・来店ポイント・連携ボーナス等）
- 利用条件（最低利用ポイント・利用単位）
- 失効条件（有効期限・失効通知タイミング）
- ログイン済み会員のポイント優遇率
- BI-SUポイントとの将来的な統合の有無

## B9. チェックイン仕様 🆕v2.1

> EP-6 に対応。レジ以外で来店を記録する機能。

### 方式候補

| 方式 | メリット | デメリット | 適合度 |
|------|---------|----------|--------|
| **店内QR読取**（会議で候補） | 導入コスト低・実装容易 | QR写真による不正 | ★★★ |
| 位置情報（ジオフェンス） | ユーザー操作不要 | 精度問題・地下店舗で不可 | ★★ |
| Bluetooth Beacon | 高精度 | ハード導入コスト | ★ |
| NFC | 確実 | 対応端末の制約 | ★ |

> **店内QRコード読取方式で方向確定**（yuka判断 2026-06-24）。不正防止・ポイント付与条件の詳細はOP-29残課題。

### 店内QR読取方式の想定フロー

1. 店舗に設置されたQRコード（店舗ごとに固定 or 日替わり💭）をユーザーがアプリで読み取り
2. アプリがチェックインAPIを呼び出し（店舗ID・日時・ユーザーID送信）
3. サーバーでバリデーション（重複チェック・時間帯チェック等）
4. チェックイン成功 → 店舗ポイント付与（EP-5連携）

### 不正防止策（💭案）

- **1日1店舗1回制限**: 同一ユーザーが同じ店舗で1日に複数回チェックインを防止
- **QRの定期更新**: 日替わり or 時間帯別でQRコードを変更し、写真共有による不正を抑止
- **サーバーサイド検証**: チェックイン頻度の異常検知（短時間に複数店舗等）
- **将来的**: 位置情報との組み合わせ検証（任意・補助的）

## B10. 多言語仕様 🆕v2.1

> EP-10 S-10.6 に対応。日本語/英語の切替に対応する。

### 対象言語

- **v1**: 日本語（デフォルト）/ English
- 将来の拡張: 中国語（簡体字/繁体字）等は後続バージョンで検討

### 切替方式

- **設定画面（SCR-509）のトグル/セレクタ**で切替
- 未ログインでも利用可能
- 選択した言語はローカルに保存（次回起動時も維持）
- OS言語設定との連動: 初回起動時はOS言語を参照して初期値を設定（日本語以外→English）

### 対象範囲

| カテゴリ | 対象 | 備考 |
|---------|------|------|
| UI文言 | ✅ ボタン・ラベル・メニュー・タブ名・エラーメッセージ | アプリ内の静的テキスト全般 |
| ブランドコンテンツ | ✅ BI-SUについて（SCR-505）・オンボーディング | 翻訳品質のレビュー必須 |
| 商品名 | 💭検討中 | 楽楽EC側の多言語対応に依存 |
| 店舗情報 | 💭検討中 | 住所・営業時間等の英語併記 |
| 通知文言 | 💭検討中 | PUSH通知の多言語化 |
| 利用規約・プライバシーポリシー | ✅ | 法務確認が必要 |

### 実装方針（💭推奨）

- Flutter標準の `intl` パッケージ + ARBファイルによるローカライズ
- 翻訳キーは `screen_section_element` 形式で命名（例: `home_status_rank_label`）
- 日付・通貨のフォーマットはロケールに応じて自動切替
- テキスト長の差異（英語は日本語より長くなりがち）に対応するレイアウト設計
