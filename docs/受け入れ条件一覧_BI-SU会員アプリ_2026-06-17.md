## 共通受け入れ条件（全ストーリーに適用）

### UI・ブランド

- [ ] BI-SUブランドのトンマナ（アイボリー×シャンパンゴールド・明朝系・上品さ）に沿ったUIである
- [ ] 表示文言がブランド表記ルールに従う（「会員」を使用し「会員様/メンバー」は不使用。ランク名はMEMBER/SILVER/GOLD/PLATINUM/DIAMONDの英字表記）
- [ ] 幅320pt級の端末（iPhone SE等）でもレイアウトが崩れない
- [ ] iOS / Android の両方で同等に動作する

### アクセシビリティ・安定性

- [ ] OSの文字サイズ設定（200%まで）でテキストが欠けず、写真背景上のテキストは十分なコントラスト（WCAG AA目安）を確保する
- [ ] 通信エラー時にアプリが固まらず、再試行またはキャッシュ表示ができる

### EC連携

- [ ] ECマイページへ遷移する操作は、再ログイン不要または最小限で、アプリへ戻った時点で表示データが再取得される

## EP-1: 認証・オンボーディング

MIRANEST Appと同じ認証フローを踏襲。メールアドレスまたは電話番号で認証コードを受け取り、認証完了後にS3上のEC顧客データと紐付けて会員情報を取得する。

### S-1.1: メールアドレスまたは電話番号で認証コードを受け取りログインする

> 会員として、いつものメールアドレスか電話番号でアプリにログインしたい。それは新しい登録をやり直したくないし、自分のステータスを失いたくないからだ。

- [ ] ログイン画面でメールアドレスまたは電話番号を入力し、「認証コードを送信」できる
- [ ] 認証コードがメール（メールアドレス入力時）またはSMS（電話番号入力時）で届く
- [ ] 受け取った認証コードを入力すると認証が完了し、S3上のEC顧客データと紐付いて会員情報（氏名・ステータス・定期・履歴）が表示される
- [ ] 認証コードには有効期限があり、期限切れ時は再送導線が表示される
- [ ] 未登録のメールアドレス/電話番号を入力した場合、EC会員登録への案内が表示される
- [ ] ログインに失敗した場合、原因（入力誤り/認証コード不一致/通信エラー等）が分かるメッセージと再試行手段がある
- [ ] 未ログイン状態では、会員証QR・個人情報・購入情報は一切表示されない

### S-1.2: 機種変更後も同じメールアドレス/電話番号で再認証して復帰する

> 会員として、スマホを買い替えても同じ認証情報で復帰したい。それは自分のステータスや購入履歴を失いたくないからだ。

- [ ] 新端末でアプリをインストールし、同じメールアドレスまたは電話番号で認証コードを受け取りログインすると、同じ会員情報・ステータスに復帰する
- [ ] 旧端末のログイン状態は、旧端末でのログアウト（S-1.4）またはセッション期限切れで無効になる
- [ ] データ移行ツール（iOS→iOS等）によるアプリ移行時も、再認証が求められる（セキュリティ上、認証済みセッションの移行は不可）

### S-1.3: 初めてアプリを開いた時、迷わず使い始める

> 会員として、アプリを初めて開いた時に何をすればいいか迷いたくない。それはスマホ操作に不慣れでも自分で進めたいからだ。

- [ ] 初回起動時、アプリの価値（会員証・ランク確認・定期管理など）が3画面以内で伝わるウェルカム画面が表示される
- [ ] ウェルカム画面からログイン画面（S-1.1）へスムーズに遷移する
- [ ] ウェルカム画面はスキップ可能で、2回目以降の起動では表示されない
- [ ] ログイン完了後、ホーム画面の主要機能（QRの出し方・タブの使い方）を簡潔にガイドするコーチマーク等がある

### S-1.4: ログアウトする

> 会員として、共有端末や機種変更時にログアウトしたい。それは自分の情報を他人に見られたくないからだ。

- [ ] 設定からログアウトでき、確認ダイアログを経て実行される
- [ ] ログアウト後、会員情報・QR・履歴等の個人情報が端末上で参照できない（キャッシュ・認証トークンも消去）
- [ ] ログアウト後はログイン画面（S-1.1）に戻り、再度メールアドレス/電話番号で認証すれば同じ会員情報に復帰できる

### S-1.5: アカウントを削除（退会）する

> 会員として、利用をやめるときにアカウントを削除したい。それは自分のデータの扱いを自分で決めたいからだ。

- [ ] アプリ内からアカウント削除の手続きに進める（App Store / Google Play の審査要件）
- [ ] 削除により何が消え、何が残るか（EC会員資格・定期契約・ポイントへの影響）が手続き前に明示される
  - ⚠ 依存: EC側の退会仕様
- [ ] 誤操作防止のため、確認ステップ（「削除する」ボタン→確認ダイアログ→認証コード再入力）を経る
- [ ] 削除完了後、端末上の全データ（キャッシュ・認証情報）が消去され、ログイン画面に戻る

## EP-2: 起動演出

ランク別ボルネオ・ズーム演出。毎回再生+スキップ確定済み。動画未取得時・reduced-motion・省電力モードのフォールバックはDoD内で定義。

### S-2.1: ランクに応じたボルネオ起動演出を毎回楽しむ

> 会員として、アプリを開くたびに自分のステータスにふさわしい特別な体験をしたい。それはBI-SUの世界（ボルネオの聖地）に近づいていく実感がほしいからだ。

- [ ] 起動のたびに、ランク別のボルネオ・ズーム演出が再生される（MEMBER=上空→島 / SILVER=島→ジャングル / GOLD=ジャングル→洞窟入口 / PLATINUM=洞窟入口→洞窟内 / DIAMOND=洞窟内→巣）
- [ ] 演出はランクの上下にかかわらず、現在の公式ステータスに基づいて表示される
- [ ] スキップボタンが常設され、いつでも演出を飛ばせる。会員証ボタン（EP-4）への直行時は演出を中断してよい
- [ ] 演出の着地ビジュアルがそのままホーム背景になり、体験が途切れない
- [ ] 演出の再生時間は約5.5秒以内とする
- [ ] OSの「視差効果を減らす」（iOS）/「アニメーションを削除」（Android）設定時は、演出を静止画1枚（着地ビジュアル）に短縮する
- [ ] 動画がまだダウンロードされていない（初回起動・CDN配信時）場合、静止画の着地ビジュアルを即表示し、動画はバックグラウンドで取得して次回起動から再生する
- [ ] バッテリー節約モード中は演出を静止画に短縮してもよい

## EP-3: ホーム・ステータス

ランク表示・ポイント残高・進捗バー・3Dロゴ。ホーム画面の中核。ポイント仕様（OP-05）とランク集計細部（OP-03）の確定待ちだが、モック値でUI先行可。

### S-3.1: ステータス・ポイント残高・特典を確認する

> 会員として、いまの自分のステータスとポイント残高、受けられる優待を確認したい。それは自分がどう遇されるのかを知り、BI-SUとの関係を実感したいからだ。

- [ ] ホームに現在の公式ステータス（MEMBER / SILVER / GOLD / PLATINUM / DIAMOND）が英字表記で表示される
- [ ] 表示されるステータスは「前年1/1〜12/31の購入累計で確定した当年ステータス」である
- [ ] 現在の還元率（100円=Npt）が表示される
- [ ] ポイント残高が表示される
  - ⚠ 依存: OP-05 ポイント残高API・有効期限の仕様
- [ ] ランク別の優待内容（バースデーギフト・シーズナルギフト・イベントご招待）が静的一覧で確認できる
  - ⚠ 依存: OP-10 特典表示の扱い
- [ ] 画面に集計ルールの注記（「ステータスは前年1年間（1/1〜12/31）のご購入累計額で決まり、当年1年間適用されます」）が表示される

### S-3.2: 翌年ステータスへの進捗を確認する

> 会員として、今年いくら購入していて、次のランクまであといくらかを知りたい。それは翌年のステータスを意識した買い物の楽しみがほしいからだ。

- [ ] 今年（1/1〜現在）の購入累計額が表示される
  - ⚠ 依存: OP-03残課題 = 税込/税抜・返品減算等の集計細部
- [ ] 次のランク閾値（30万/50万/100万/200万円）までの残額と進捗バーが表示される
- [ ] 進捗は「翌年ステータスの見込み」であることが分かる表現になっている（現ステータスとの混同を防ぐ）
- [ ] 当年累計が現ステータスの閾値未満でも、マイナス等の不自然な表示にならない
- [ ] DIAMOND会員には進捗の代わりに最上位への感謝メッセージが表示される

### S-3.3: 年次のステータス切替わりを理解する

> 会員として、年が変わって自分のステータスがどうなったかを正しく知りたい。それは新しい1年の優待と目標を、誤解なく把握したいからだ。

- [ ] 1/1以降にアプリを開くと、新しい当年ステータスと、その根拠（前年の購入累計）が確認できる
- [ ] 進捗バー・当年累計は0円から再スタートし、前年の数値と混在しない
- [ ] ステータスが下がった場合も、否定的な文言を使わず、現ステータス準拠の演出・表示で迎える
- [ ] 切替時期（1/1）と適用期間が画面の注記で確認できる

### S-3.4: 3Dロゴエンブレムがランクに応じて表示される

> 会員として、自分のランクを象徴する美しいロゴを見たい。それはBI-SUの世界観に包まれている実感が持てるからだ。

- [ ] ホーム画面にBI-SUロゴの3D表現が表示される
- [ ] ランクによってロゴの色味が切り替わる（MEMBER=シャンパン / SILVER=シルバー / GOLD=ゴールド / PLATINUM=青系 / DIAMOND=深紅系）
- [ ] 3D描画が不可能な端末（GPU制限等）ではフラットなSVGロゴにフォールバックする
- [ ] ロゴの3D形状は公式ベクターデータを基にする（入手前は手作業トレース版で代替）

## EP-4: 会員証QR

QR表示・輝度制御・オフライン保証・スマレジ連携・代替手段。アプリの第一利用シーン「店舗での会員証提示」を、会員とスタッフの両側から成立させる。

### S-4.1: レジで素早く会員証を提示する

> 会員として、レジで待たせずに会員証を提示したい。それは後ろのお客様を気にせず、スムーズに会計を済ませたいからだ。

- [ ] ホームのファーストビューから1タップで会員証QRを表示できる
- [ ] アプリのコールドスタートから、スキップ操作1回を含めてQR表示まで3秒以内（自動計測: アイコンタップ起点〜QR描画完了）
- [ ] 起動演出中もスキップ・会員証ボタンが常時表示され、タップから0.3秒以内に反応する
- [ ] QR表示中は画面輝度が端末の最大値に引き上げられる
- [ ] QRには会員IDが含まれ、会員ランクは含まれない（ランクはスマレジ/EC側データを正とする）

### S-4.2: 電波なし・メンテ中でも会員証を使う

> 会員として、地下の店舗（GINZA SIX B1F等）で電波がなくても、システムメンテナンス中でも会員証を提示したい。それはレジの前で「表示できない」と慌てたくないからだ。

- [ ] 機内モード・圏外の状態でも、QRコードが表示される（事前生成・キャッシュ）
- [ ] オフライン時に表示されるコードでも、店舗で会員特定が可能である
  - ⚠ 依存: OP-01 トークン方式の採否
- [ ] サーバメンテナンス・API障害中も、会員証QRはキャッシュから表示できる
- [ ] オフライン・メンテ中はその旨が表示され、最新でない可能性のあるデータ（ポイント等）と区別できる

### S-4.3: （スタッフ）QRからスマレジに会員ID取込

> 店舗スタッフとして、会員のQRをその場で読み取って会員IDをスマレジに取り込みたい。それは入力の手間とミスをなくし、接客に集中したいからだ。

- [ ] アプリが表示するQRを、スマレジ（または接続スキャナ）で読み取ると会員IDが転記される
- [ ] QRのコード体系がスマレジの読取仕様と一致している
  - ⚠ 依存: スマレジ読取仕様の確認
- [ ] 読み取り→会員特定までがレジ操作の流れの中で完結する（別端末・別アプリへの持ち替えがない）
- [ ] 購入データの履歴・累計への反映は S-6.2 で受け入れる（本ストーリーは読み取り・転記まで）

### S-4.4: QRが使えない時に代替手段で本人確認する

> 会員として、スマホの電池切れやアプリ不調のときでも会員として扱ってほしい。それはせっかく店舗まで来たのに優待を受けられないのは悲しいからだ。

- [ ] QR画面から会員IDをテキストで確認できる（プライバシー配慮のためマスク表示+タップで開示）
- [ ] QRの生成に失敗した場合、自動で会員IDテキスト+バーコード等の代替表示に切り替わる
- [ ] スマホ自体を出せない場合の店頭照会手順（氏名+電話番号等）が店舗オペレーションとして定義されている
  - ⚠ 依存: 店舗手順書

## EP-5: 定期情報

定期コースの閲覧専用ミラー。変更操作はECマイページへ橋渡し。EC顧客のユーザビリティ向上＝アプリの主目的のひとつ。

### S-5.1: 次回お届けを確認する

> 会員として、次の定期便がいつ届くかをすぐ確認したい。それは受け取りの予定を立て、飲み忘れなく続けたいからだ。

- [ ] 利用中の各定期コースに、次回お届け予定日が表示される
- [ ] お届けサイクル（30日ごと等）が表示される
- [ ] 表示はECシステムの最新データと一致する
  - ⚠ 依存: OP-24 EC API。データの更新タイミングが画面で確認できる

### S-5.2: 定期コースの内容を確認する

> 会員として、自分がどの定期コースを、いくらで、何個受け取っているか確認したい。それは家計の把握と、コース内容の思い違い防止のためだ。

- [ ] 各コースに商品画像・商品名・個数・金額（税込）が表示される
- [ ] 複数の定期コースを契約している場合、すべて一覧で確認できる
- [ ] ステータス（継続中など）がひと目で分かる
- [ ] 定期コースが0件の場合、空状態の案内と定期コースの紹介（ECへの導線）が表示される

### S-5.3: 定期の変更手続きへ進む（ECマイページ誘導）

> 会員として、お届け日の変更やスキップをしたいとき、アプリから迷わず手続きに進みたい。それは電話をかけずに自分のタイミングで済ませたいからだ。

- [ ] 各定期コースから「変更・スキップ・解約のお手続き」への導線があり、ECマイページの該当ページへ遷移できる
- [ ] 次回お届けの変更締切日（何日前まで変更可能か）が表示される
  - ⚠ 依存: EC仕様
- [ ] 手続き後にアプリへ戻ると、変更が反映された表示になる

### S-5.4: 解約済みの定期を確認する

> 会員として、以前利用していた定期コースを確認したい。それは「また再開しようか」を考える材料にしたいからだ。

- [ ] 解約済みコースが利用中と区別されて表示される（最終お届け日付き）
- [ ] 解約後12ヶ月以内のコースが表示される
  - ❗ 要決定: 表示期間。候補=6/12/24ヶ月
- [ ] 解約済みコースから商品の再開・再購入導線（ECへの遷移）がある

### S-5.5: 定期の異常（決済不備等）に気づく

> 会員として、定期便が止まりそうなとき（カード期限切れ等）に早く気づきたい。それは「届くはずのものが届かない」を避けたいからだ。

- [ ] 決済エラーや一時停止などの異常状態が、通常の「継続中」と明確に区別して表示される
  - ⚠ 依存: OP-24 ECの定期ステータス仕様
- [ ] 異常がある場合、ホームと定期情報タブにアラートバッジが表示され、該当コースが一覧の最上部に表示される
- [ ] 異常状態のコースには対処方法（ECマイページでのカード更新等）への導線が表示される
- [ ] 重要なお知らせ（決済不備等）はPUSH通知でも届く（S-8.2の「重要なお知らせ」カテゴリ=常時ON）
  - ⚠ 依存: OP-09 PUSH配信基盤

## EP-6: 購入履歴

ネット注文・店頭購入を横断した購入の記録。会員の「自分の買い物の控え」であり、CS問い合わせの自己解決装置。

### S-6.1: 過去の注文を確認する

> 会員として、過去に何をいつ買ったか確認したい。それは「前に買ったのはどれだったか」を思い出して、次の買い物や問い合わせに使いたいからだ。

- [ ] 注文一覧に、各注文の日付・注文番号・商品画像・商品名・数量・金額（税込）が表示される
- [ ] 表示日付の定義: ネット注文=出荷日、店頭購入=購入日。一覧はこの日付の降順で並ぶ
- [ ] 注文の種別（定期購入 / 通常購入）と経路（ネット注文 / 店頭購入）が表示される
- [ ] 1注文に複数商品が含まれる場合、全商品が確認できる
- [ ] 過去24ヶ月分を対象とし、20件ずつ追加読み込みできる
  - ❗ 要決定: 範囲。候補=24ヶ月/全期間
- [ ] 履歴が0件の場合、空状態の案内が表示される

### S-6.2: 店頭購入も履歴で確認する

> 会員として、店舗で買ったものもアプリの履歴で見たい。それはネットと店舗を分けずに、BI-SUとの買い物をひとつの記録として持ちたいからだ。

- [ ] 会員証を提示して店頭購入した注文が履歴に表示される
  - ⚠ 依存: スマレジ→EC連携
- [ ] 店頭購入には購入店舗名が表示される
- [ ] 店頭購入が年間購入累計（ステータス進捗）に反映される（包含が確定した場合）
  - ⚠ 依存: OP-03残課題 店頭購入のランク集計包含
- [ ] 店頭購入が履歴に反映されるまでの目安時間が定義され、ユーザーに分かる

### S-6.3: 目的の注文を素早く見つける

> 会員として、目当ての注文をすぐ見つけたい。それは履歴が増えても探すのに時間をかけたくないからだ。

- [ ] 種別（定期/通常）・経路（ネット/店頭）で一覧を絞り込める
- [ ] 一覧はS-6.1の表示日付の降順で並び、スクロールで遡れる
- [ ] 100件読み込み済みの状態でも、スクロールにフリーズ・コマ落ちが発生しない
- [ ] 追加読み込み中は1秒以内にローディング表示が出る

### S-6.4: 問い合わせ時に注文番号を示す

> 会員として、問い合わせのときに注文番号をすぐ伝えたい。それはCSとのやりとりを早く正確に済ませたいからだ。

- [ ] 各注文の注文番号が表示され、長押し等でコピーできる
- [ ] 注文の詳細表示から問い合わせ導線（S-8.4と同じ窓口）に進める
- [ ] アプリの表示とECの正式記録が食い違った場合の正本がECであることが利用案内に明記される

### S-6.5: 注文のいまの状態を知る

> 会員として、注文した商品がいまどういう状態か（出荷前・出荷済み・返品済み）を知りたい。それは「注文したのに履歴に出ない」「返したのに載っている」という不安を持ちたくないからだ。

- [ ] 出荷前の注文が「出荷準備中」等のステータス付きで表示される。表示対象外とする場合は、その旨の案内文言が一覧にある
  - ⚠ 依存: OP-15 注文ステータスの扱い
- [ ] 返品・キャンセルされた注文が、通常の注文と区別されて表示される
- [ ] 表示する注文ステータスの列挙が、ECシステムの実際のステータス値と一致している
  - ⚠ 依存: OP-24 EC仕様

## EP-7: ストアーズ

会員を店舗へ案内し、迷わせず到着させる。店舗送客=アプリの主目的のひとつ。

### S-7.1: 近くの店舗を探す

> 会員として、自分の近くにBI-SUの店舗があるか知りたい。それは商品を実際に手に取り、相談しながら選びたいからだ。

- [ ] 店舗がリスト表示され、各店舗に外観写真・店舗名・現在地からの距離・住所が表示される
- [ ] 位置情報許可時は現在地から近い順に並ぶ。未許可時は固定順（直営店→取扱店→期間限定）で表示される
- [ ] 距離は実際の現在地から計算される（位置情報許可時）
- [ ] 店舗区分（直営店 / 取扱店 / 期間限定）がひと目で区別できる
- [ ] 店舗データの追加・変更がアプリ更新なしで反映できる仕組みである
  - ⚠ 依存: OP-21 店舗データの更新運用

### S-7.2: 地図で店舗の位置をつかむ

> 会員として、店舗の位置関係を地図で見たい。それは旅行先や外出先で「行ける場所か」を直感的に判断したいからだ。

- [ ] リスト⇔マップをワンタップで切り替えられる
- [ ] 地図上のピンから店舗を選び、詳細へ進める
- [ ] 複数店舗が近接するエリア（東京・京都）でも個別の店舗を選択できる

### S-7.3: 店舗の詳細を見て迷わず行く

> 会員として、行くと決めた店舗の場所・行き方・営業時間を知りたい。それは「着いたのに閉まっていた」「館内で見つからない」を避けたいからだ。

- [ ] 店舗詳細に外観写真・説明文・営業時間・定休日・電話番号（公開店舗のみ）・住所・アクセスが表示される
- [ ] 商業施設内の店舗は、フロアと館内のおおよその位置が分かる案内（フロアマップ等）がある
- [ ] 「経路を見る」からOSの地図アプリが起動し、店舗への経路案内が始まる
- [ ] 営業時間・定休日は公式情報と一致している
  - ⚠ 依存: OP-17 定期点検で鮮度を維持

### S-7.4: 期間限定ストアを知る

> 会員として、近くで開催されるポップアップストアを見逃したくない。それは期間限定の機会に実物を見たいからだ。

- [ ] 期間限定店舗は「予告（開催前）」「開催中」「終了」の3状態で管理される
  - ⚠ 依存: OP-21 店舗データに開催前状態を追加
- [ ] 開催期間が明示され、開催中であることがひと目で分かる
- [ ] 終了後7日間は「終了しました」表示で一覧に残り、その後自動的に一覧から消える

### S-7.5: 位置情報を許可せずに店舗を探す

> 会員として、位置情報を渡さなくても店舗一覧は使いたい。それはプライバシーは自分でコントロールしたいからだ。

- [ ] 位置情報の許可は店舗機能の初回利用時に、用途の説明付きで求められる（起動直後に求めない）
- [ ] 許可しない場合も店舗一覧・地図・詳細は利用できる（距離のみ非表示）
- [ ] 後から設定で許可した場合、距離表示が有効になる

- [ ] 状態パターンは共通Widgetとして提供され、各画面が個別実装しない
